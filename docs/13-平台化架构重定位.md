# AegisCDN 平台化架构重定位(基于成熟组件的商用高防 CDN SaaS)

版本 v2.0 · 2026-05-22 · **方向修正**:停止重造底层网络轮子,聚焦商业化 SaaS;底层全部基于成熟组件。

> 本文是项目的新北极星。它**取代** docs/09、docs/12 中"自研 WAF/CC 执行引擎"的部分定位:
> 底层执行交给成熟组件,我们做**控制面 + 业务 SaaS + 编排/可视化**。

---

## 核心原则:不重复造的轮子(用成熟组件)

| 能力 | ❌ 不自研 | ✅ 采用成熟组件 |
| --- | --- | --- |
| HTTP 服务器 / 反向代理 / 缓存 | 自写代理/缓存核心 | **OpenResty / Nginx**(proxy / proxy_cache) |
| TLS 协议 / 证书 | 自写 TLS | **OpenSSL + OpenResty SNI + ACME(acme.sh/lego)** |
| **WAF 底层执行引擎** | 自写正则规则引擎(原 `waf.lua`) | **Coraza(OWASP CRS)** 或 **ModSecurity v3 + OWASP CRS** |
| CC / 限流执行 | 自写滑动窗口算法 | **lua-resty-limit-traffic** + Redis(成熟库) |
| TCP 协议栈 | — | **操作系统内核**(必要时 eBPF/XDP,基础设施层) |
| 缓存 / 会话 / 队列 | — | **Redis / Redis Cluster** |
| 日志分析 | — | **ClickHouse**(+ 后期 Kafka) |
| 监控 | 自写 | **Prometheus + Grafana**(+ nginx-exporter / node-exporter) |
| 业务库 | — | **PostgreSQL** |

**我们自己做的 = 上层价值**:多租户 SaaS、计费、域名接入编排、规则**配置编译与下发**、风控**策略编排**、
攻击**可视化**、代理商、工单、开放平台。底层引擎只"配置 + 调用",不"重写"。

## 4. 需要重构 / 替换的模块

| 现状(自写) | 问题 | 重构为 |
| --- | --- | --- |
| `edge/.../waf.lua` 正则规则执行引擎 + 内置签名 | 重造 WAF 执行引擎 | **Coraza(OWASP CRS)/ ModSecurity v3 + CRS**;`config-compiler` 改为生成 CRS/SecRule + 自定义规则 |
| `ratelimit.lua` 自写滑动窗口/令牌桶 | 重造限流算法 | **lua-resty-limit-traffic / limit.count** + Redis(成熟库),平台只配阈值 |
| `bot.lua` / `risk.lua` / `challenge.lua` / `reputation.lua` | 不算"底层引擎",但定位要清 | 收敛为**风控编排层**:采集信号 + 调用成熟原语 + 决策动作(挑战/限流/调 WAF),不写协议 |
| `config-compiler` 输出 OpenResty 私有结构 | 与自写引擎耦合 | 输出**标准件配置**:Nginx vhost + Coraza/CRS 配置 + limit 配置 + 证书,经 agent 下发 reload |
| docs/09 / docs/12 的"自研引擎"叙事 | 与新方向冲突 | 本文(13)为准:底层用成熟件,我们做编排 |

> 迁移策略:**绞杀者模式(Strangler)**。OpenResty 仍是数据面入口;把 `access_by_lua` 里的
> WAF 执行替换为调用 Coraza;限流替换为 lua-resty-limit-traffic;风控编排层保留并精简。
> 业务面(NestJS/前端/DB)**零改动**,只是 config-compiler 的输出目标变了。

## 5. 新目录结构(模块化,可微服务拆分)

```
aegis-cdn/
├── apps/
│   ├── web/                       # Next.js 前端三端(官网/控制台/管理后台)
│   └── api/                       # NestJS 模块化单体(每模块=未来一个微服务)
│       └── src/modules/
│           ├── identity/          # auth + tenant + rbac + api-key
│           ├── billing/           # plan + subscription + order + wallet
│           ├── payment/           # 支付适配器(已就绪)
│           ├── provisioning/      # domain + ssl + config-compile + deploy
│           ├── security-policy/   # waf 规则 + cc 策略(配置,不执行)
│           ├── edge-control/      # node 注册/心跳/下发/灰度
│           ├── analytics/         # stats + 日志查询(ClickHouse)
│           ├── reseller/          # 代理商
│           ├── ticketing/         # 工单
│           └── open-api/          # 开放平台(API Key 鉴权 + 限频)
├── edge/
│   ├── openresty/                 # 反代/缓存/TLS(成熟件)
│   ├── coraza/                    # WAF 执行(OWASP CRS + 自定义规则,平台下发)
│   ├── agent/                     # Go:拉配置 + 推日志 + 上报指标
│   └── exporters/                 # nginx-exporter 配置(Prometheus)
├── packages/shared/               # 共享 DTO / 类型 / 错误码
├── infra/
│   ├── compose/                   # docker-compose(dev/prod)
│   ├── prometheus/  grafana/      # 监控配置 + 仪表盘
│   └── k8s/                       # (后期)Helm/manifests
└── docs/
```

**模块化铁律**:每个 module 只暴露 service 接口;跨模块调用走接口或事件,**禁止直接读对方表**;
DTO 在 packages/shared;这样任一 module 可原样抽成独立微服务。

## 7. 服务拆分方案(模块化单体 → 微服务)

**现在:模块化单体**(NestJS,一个进程,多 module),开发/部署简单,边界清晰。
**未来:按 bounded context 拆**(先拆读写最重的):

| 优先级 | 微服务 | 触发拆分的信号 |
| --- | --- | --- |
| 1 | **analytics-svc**(日志/统计) | ClickHouse 查询压力大、独立扩展 |
| 2 | **billing+payment-svc** | 计费/对账独立、合规隔离 |
| 3 | **edge-control-svc** | 节点规模大、下发频繁 |
| 4 | identity / provisioning / security-policy / reseller / ticketing / open-api | 按需 |

**通信**:同步用 REST(内部 gRPC 可选);异步用 **Kafka 事件**(order.paid / domain.approved /
node.flagged …)解耦。各服务独立 DB schema 或独立库,通过事件最终一致。

## 8. 节点通信方案(控制面 ↔ 边缘节点)

```
注册:    节点装 agent → POST /edge/register(node token)→ 控制面登记 Node
心跳:    agent 每 5–10s POST /edge/heartbeat(状态/负载/已应用 configVersion)
配置下发: 控制面写 Redis(版本化)+ PUBLISH 通知 → agent 拉取增量 → 渲染 Nginx/Coraza/limit
          → nginx -t → reload(失败保留旧版回报 failed,可灰度/回滚)
封禁广播: 全局封禁/信誉 → Redis pub/sub → 各节点即时生效(跨节点)
日志上报: OpenResty JSON 日志 → agent 批量 → (后期 Kafka)→ ClickHouse
指标上报: nginx-exporter / node-exporter → Prometheus 拉取
```
**安全**:agent ↔ 控制面用 **mTLS 或 node token + HMAC 签名**;控制面内部端口(Redis/CH/PG)
**仅对节点 IP 白名单**;回源对源站用签名头 + 源站只放行边缘 IP。

```

        │
        ▼  agent 批量(后期前置 Kafka 削峰 + 多消费者)
   ┌────────────┬─────────────────────────────┐
   ▼            ▼                             ▼

(明细+物化     │                               
 视图聚合)     ▼
   │        Grafana(运维大盘) + 平台后台(/admin/security 业务大盘)
   ▼
```

## 10. 未来高并发优化路线(分阶段)

| 阶段 | 重点 | 产出 |
| --- | --- | --- |
| **P1 平台化(当前)** | 底层换成熟件(Coraza/CRS、lua-resty-limit、proxy_cache);补齐 15 项 SaaS 功能;单区 | 可商用单区 SaaS |
| **P2 多节点 + 缓存层** | 多边缘节点 + DNS/GeoDNS 调度 + 健康摘除;**Redis Cluster**;PG 读副本 | 多区加速 + 高可用 |
| **P3 日志规模化** | **Kafka** 前置日志管道 + **ClickHouse 集群**(分片/副本);Prometheus 联邦 | 海量日志可观测 |
| **P4 微服务拆分** | 先拆 analytics / billing;Kafka 事件解耦;API 网关 + 服务发现 | 团队/容量独立扩展 |
| **P5 基础设施级防护** | 上游**云清洗 / Anycast** 抗 L3/L4;裸金属边缘上 **eBPF/XDP**;多区控制面 HA | 容量级高防 |

> 性能根基不变:**边缘无状态(状态在 Redis)、热路径本地命中、Redis 分片、fail-open、内核调优**;
> 加速核心(缓存/反代/TLS)始终由 **OpenResty** 等成熟件承担,我们不碰协议栈。
